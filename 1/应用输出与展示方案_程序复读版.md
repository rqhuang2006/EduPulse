# 应用输出与展示方案

## 8.1 文档依据与说明

本节内容依据当前程序的实际实现进行整理，并非脱离代码状态的概念性方案说明。本文对应的核心文件包括 `app.py`、`src/frontend/data.py`、`src/frontend/components.py`、`src/frontend/pages.py` 和 `src/frontend/state.py`。其中，`app.py` 负责系统入口、数据加载与页面调度；`data.py` 负责多源产物读取、主表补齐、展示字段清洗和预测入参标准化；`components.py` 负责整体视觉样式、侧边栏、指标卡片和表格展示；`pages.py` 负责各页面功能实现；`state.py` 负责角色状态、页面状态、筛选条件和学生焦点的维护。

从当前程序结构看，系统采用 Streamlit 作为前端承载框架，整体为单应用多页面结构。系统启动后会依次执行样式注入、数据准备、会话状态初始化、角色工作区选择、侧边栏渲染和页面内容渲染。当前程序已经形成较为完整的教师/管理端与学生自助端双视图结构，但需要如实说明的是：管理端与教师端在代码中并未拆分成两套独立菜单，而是共用“教师/管理端”这一套工作区；学生端则独立为“学生自助端”，包含个人主页、状态自评和 AI 树洞三个页面。因此，以下内容严格以程序现状为准，不将尚未独立拆分的能力写成已经完全实现的系统结构。

## 8.2 展示对象

根据当前程序实现，系统的主要展示对象包括教师/管理端用户、学生自助端用户以及项目展示与答辩场景下的汇报对象三类。

教师/管理端是当前程序中的核心工作区，既服务于学校或学院层面的管理人员，也服务于辅导员、班主任和任课教师等一线教育工作者。由于这两类角色当前在代码中共用同一套导航和页面结构，系统主要通过页面内容的层级差异来覆盖管理分析与教师研判两种使用场景。例如，总览工作台、群体画像、成果总览和动态建模展示更偏向管理分析与成果汇报；学生档案、链路追踪、精准预警干预和 AI 辅导员助手更偏向教师个案分析与跟进场景。

学生自助端是程序中已经独立建立的另一类展示对象，当前包含“我的成长主页”“状态自评”和“AI 树洞”三个页面，主要面向学生本人，提供相对轻量的个人状态查看、自评反馈和情绪支持式交互。这部分功能在当前程序中已经具备独立入口和独立页面，但整体信息粒度明显低于教师/管理端，符合“学生端以自我感知和辅助反馈为主”的设计取向。

此外，从程序的页面组织方式看，系统还兼顾项目答辩、成果展示和阶段验收等汇报场景。成果总览页和动态建模展示页不仅服务于日常操作，也承担了集中说明系统建设情况、成果结构与技术验收结果的作用，因此同样构成程序中的重要展示对象。

## 8.3 展示内容

从当前页面实现来看，系统已经覆盖整体风险总览、群体画像、个体档案、重点对象清单、分析成果展示、干预建议、问答辅助内容以及动态建模验收结果等多个层面。

在整体层面，程序通过“学校端总览工作台”页面集中展示学生总数、高风险人数、平均综合风险和主导维度等汇总指标，并在同一页面中提供重点学生清单、行为模式分布图和三维风险区间图。这一页面同时内嵌“校园智能助手”条带，支持围绕整体风险情况、重点学生和干预建议做快捷提问。因此，总览页在当前程序中承担的是“整体感知 + 初步研判 + 快速问答入口”的复合功能，而不是单一静态看板。

在群体层面，程序通过“群体画像”页面展示当前筛选样本的主要行为模式、群体主导机制和三维风险区间。页面既提供样本数、主要模式和主导机制等汇总指标，也通过群体模式图、三维风险区间图和模式表来呈现学生群体的结构特征与共性问题。从实际代码实现看，该页面已经覆盖了群体分类、群体描述、群体风险结构和群体解释说明等核心内容。

在个体层面，程序通过“学生数字档案”页面输出学生级分析结果。当前页面实际展示内容包括学生编号、综合风险、主导维度、优先级、学习风险、生活风险、运动风险、三维风险图、档案标签与建议、三维解释以及多 Agent 依据。其中，“三维解释”已经被整理为结构化表格，围绕行为证据、归因机制和环境支持三项内容输出核心结论、数据依据、解释视角和建议动作；“多 Agent 依据”则保留风险判断、行为证据、机制解释和干预输出等模块内容。因此，个体档案已经形成了“风险结果 + 解释结构 + 支持建议 + 多模块依据”的综合输出方式。

在重点对象识别与预警方面，程序通过“精准预警干预”页面提供阈值调节、预警人数统计、P1 人数统计和预警学生清单。页面能够根据当前阈值动态筛选高风险学生，并输出风险等级、综合风险、主导维度、行为模式和优先级等内容，用于支撑教师快速形成重点关注名单。与总览页的重点学生清单相比，该页面更偏向正式的预警筛查口径。

在成果展示方面，程序已通过“成果总览”页面形成较完整的展示结构。当前该页包含“任务概览”“用户期望”“成果清单”“学生成果”和“技术指标”五个子部分。页面顶部展示用户期望覆盖、成果类型数量、学生成果数量和最佳 AUC 等概览指标；“成果清单”部分展示成果分组与成果明细；“学生成果”部分集中展示 10 名学生分析成果；“技术指标”部分展示指标验收条目并支持跳转到对应页面。因此，程序目前已经把成果展示从单纯页面罗列提升为结构化成果目录。

在链路解释方面，程序通过“学生链路追踪”页面展示具体学生从三域风险输出到融合判断、解释线索、机制判断、模式标签和干预输出的完整链路。该页面支持切换学生对象，并将三域原始输出、融合结果、解释线索、机制判断、模式标签、链路摘要和干预输出组织在同一页面中。尤其是链路摘要与干预输出部分，已经通过自然语言进行了人话化转换，不再直接暴露原始字段名和模型中间产物，因此具备较好的解释性与展示性。

在问答辅助方面，程序当前有两类实现。一类是总览页内嵌的“校园智能助手”条带，它围绕全校情况、重点学生和干预建议提供快捷追问能力；另一类是独立的“AI 辅导员助手”页面，支持默认学生选择、自由提问和会话记录保留。需要客观说明的是，当前程序中的问答能力主要基于本地已有主表、报告、模式统计和 harness 状态拼接生成，而非独立的大模型在线推理逻辑。因此，在报告中更适合表述为“基于现有分析结果的问答辅助能力”。

在技术展示方面，程序通过“动态建模展示”页面提供“运行与验收”“建模架构”“工程明细”“离线研发”四个子部分。运行与验收部分包含实时运行建模面板、Harness 三域验收卡片和闭环概览；建模架构部分展示多源数据范围和多 Agent 输出覆盖；工程明细部分展示工程产物同步、三域数据同步和主线验收明细；离线研发部分展示分类与回归任务的离线建模记录。因此，该页面不仅用于技术人员查看运行结果，也承担了项目技术说明和验收支撑的展示职责。

学生端当前展示内容相对精简，但已经具备明确结构。“我的成长主页”展示学生本人的综合风险、重点关注维度、建议优先动作和成长摘要；“状态自评”支持学生通过表单输入压力、睡眠、支持程度、运动状态和情绪稳定度，并生成自评状态、负荷指数和建议文本；“AI 树洞”支持学生输入近期感受，并返回安抚式、建议式的自然语言反馈。由此可见，学生端已初步形成“查看自身状态 + 自评反馈 + 情绪支持”的闭环，但仍属于轻量使用场景。

## 8.4 展示形式

从程序实际实现来看，系统的展示形式主要包括网页页面、可视化看板、结构化成果表、报告文本、学生级说明内容和交互式卡片式展示等几类。

网页页面是系统最核心的展示载体。程序通过统一的侧边栏和角色切换入口组织所有页面，并在教师/管理端与学生端之间建立不同导航结构。页面内部广泛使用容器、列布局、卡片式摘要、结构化表格和图表组件，使各类信息能够在一屏之内完成分层呈现。

可视化看板主要体现在总览工作台、群体画像和动态建模展示等页面。程序中已经实现的图表类型包括行为模式分布图、三维风险区间图、三维风险柱状图等，这些图表均经过统一主题样式处理，作为汇总看板的重要组成部分。结构化成果表则贯穿于重点学生清单、模式表、链路原始输出表、机制判断表、技术指标验收表和多 Agent 输出表等多个位置，是当前程序中最主要的规范化信息承载方式。

报告文本和学生级说明内容同样是当前程序的重要展示形式。无论是在学生数字档案中的系统生成档案、链路追踪页中的链路摘要与干预输出，还是 AI 助手中的问答回复，程序都已经将部分分析结果转化为自然语言文本，用于增强结果的可读性与业务解释性。这说明当前程序采用的是“指标卡 + 图表 + 表格 + 文本说明”并行的复合表达形式，而不是单纯依赖图表或表格进行展示。

## 8.5 交互能力

从交互实现角度看，当前程序已经具备较完整的页面切换、条件筛选、对象下钻、快捷跳转、智能问答、自评输入与动态刷新能力。

页面切换方面，系统当前支持通过顶部角色切换控件在“教师/管理端”和“学生自助端”之间切换，并通过侧边栏单选导航在各自工作区内选择页面。教师/管理端当前包含总览工作台、成果总览、动态建模展示、群体画像、学生档案、预警干预、新增预测、AI 助手和链路追踪等页面；学生端当前包含我的成长主页、状态自评和 AI 树洞三页。

条件筛选方面，侧边栏在教师/管理端提供全局筛选能力，支持按风险等级、主导维度、行为模式以及综合风险下限进行过滤。筛选结果会同步作用于总览、群体、成果和预警等页面，从而形成统一的数据上下文。

对象下钻与快捷跳转方面，程序已实现多种页面联动逻辑。例如，成果总览中的指标查看按钮可以跳转到学生档案、群体画像、动态建模展示等对应页面；学生成果区和样例入口可以直接进入学生档案或链路追踪；链路追踪页面支持直接切换学生对象；侧边栏中的学生焦点也会影响个体页展示。这些实现使程序具备较好的“从宏观到微观、从成果到详情”的下钻能力。

智能问答方面，当前程序已形成两套交互入口：一套是总览页内嵌的快捷追问条带，一套是独立 AI 助手页中的对话输入框。前者更适合演示场景中的快速发问，后者更适合围绕指定学生或筛选后的对象范围持续追问。虽然其底层逻辑仍是基于现有分析结果和规则化文本拼接，但从前端交互角度看，已经能够实现快捷提问、历史记录保留和对象关联问答等功能。

自评输入与反馈方面，学生端“状态自评”页面提供了表单式输入能力，学生可以通过滑块输入多项主观状态信息并生成即时反馈结果；“AI 树洞”页面则提供文本输入和安抚式反馈。这说明学生端不仅是静态信息展示，也已经包含了主动输入与反馈回路。

动态刷新与运行方面，动态建模展示页当前提供“触发三域运行”和“刷新结果”按钮，说明程序已经预留了前端发起一次实时运行并查看最新结果的能力。不过需要客观说明，这部分能力依赖 harness 运行接口和相关运行环境是否就绪，因此在文档中更适合表述为“已实现动态触发入口与结果刷新机制”，而不是无条件保证任何环境下都能即时执行。

## 8.6 当前实现特征与边界说明

结合程序现状，当前系统已经形成较完整的展示骨架和较丰富的前端交互能力，尤其在教师/管理端页面组织、成果展示结构、链路追踪表达和学生级档案说明方面完成度较高。同时，数据准备层已经具备对主表风险值、解释字段和部分缺失展示字段的清洗与补齐能力，这为页面稳定展示提供了重要支撑。

但从严格意义上讲，当前程序也存在一些需要在报告中如实说明的边界。第一，管理端与教师端目前仍为同一工作区，尚未拆分成两套完全不同的菜单和权限体系；第二，AI 问答能力当前主要基于已有分析结果和本地逻辑生成回复，尚不宜表述为完全开放式智能对话系统；第三，动态建模展示页虽然已具备运行入口和验收展示，但其完整效果仍依赖外部 harness、模型产物和运行环境。将这些边界在文档中说明清楚，更有助于使报告与程序现状保持一致，也更符合正式项目报告对真实性和可核查性的要求。
